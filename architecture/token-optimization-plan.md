## 一、问题重新定位：输入成本是真正的Token黑洞

### 1.1 消耗结构重新分析

**之前的错误假设：** exec输出占token消耗的66.1%
**实际账单数据：** 输入成本 >> 输出成本

| 消耗类型 | 占比估计 | 问题根源 | 优化潜力 |
|---------|---------|---------|---------|
| **心跳上下文加载** | ~80-90% | 每次心跳加载：HEARTBEAT/MEMORY/SOUL/AGENTS.md + 历史对话 | 极高 |
| **Agent间通信** | ~5-10% | 双向消息传递 | 中高 |
| **exec输出** | ~5-10% | 大输出未截断 | 中高 |
| **模型推理** | ~5% | 大模型过度配置 | 中 |

### 1.2 心跳机制的放大效应

**放大公式：**
```
总输入消耗 = Agent数量 × 心跳频率 × 每次心跳加载上下文大小
```
**当前数据：**
- 11个Agent × 30分钟 = 每30分钟22次完整上下文调用（输入+输出）
- 每次心跳加载：~200行HEARTBEAT.md + ~500行其他文件 + 历史对话
- 每日心跳次数：528次（11×48）

**结论：** 优化输入上下文加载的ROI远高于优化exec输出！

**预计输入节省：** 立即减少75%心跳输入！

#### 2. ❗ 心跳频率分级调整（输入优化核心）

| Agent类型 | 当前频率 | 优化后 | **输入节省** | 优先级 |

| **核心**（阿尔法） | 30分钟 | 30分钟 | — | 必须保持 |

**预计输入节省：** 每次心跳减少10-20%上下文大小

**预计输入节省：** 每次心跳减少15-25%历史对话加载

### Phase 3：架构优化（ROI长期，2-4周）

#### 9. 🤝 Agent职责合并的专业意见

**问题澄清：**
- **Agent合并**：减少Agent数量，合并工作空间
- **职责合并**：保持Agent数量，但优化职责划分

**专业建议：**

**方案A：职责优化（推荐）**
```
研究+数据：拉姆达（研究）+ 德尔塔（数据）→ 保持独立，加强协作
技能+自动化：伽马（技能）+ 泽塔（自动化）→ 保持独立，接口标准化
知识+沟通：艾塔（知识）+ 西塔（沟通）→ 保持独立，流程打通
```

**优势：**
- 保持各Agent专业性和响应速度
- 无需大规模迁移和数据合并
- 通过接口标准化实现1+1>2

**方案B：Agent数量减少（风险较高）**
- 合并拉姆达+德尔塔 → 研究数据分析一体化
- 但可能造成：上下文混乱、响应延迟、技能冲突

**建议：** 先实施**方案A（职责优化）**，效果不理想再考虑方案B。

**目标：** 非核心Agent不常驻，事件驱动启动，减少40-60%空转时间

| 层级 | Agent | 模式 |
|------|-------|------|

- 需要预热机制（启动延迟控制在30秒内）

**计划执行：** 3/22-3/23（基础优化完成后再做）

### 8.2 心跳事件驱动化

**目标：** 非核心Agent心跳只在有事件时触发，没事不跑，空闲时零token消耗

**现状：** 不管有没有事，每30/60/120分钟准时跑一遍
**改为：** 有inbox消息/有cron触发才跑，没事不跑

**实施要点：**
- 核心Agent（阿尔法）保持轮询心跳
- 功能Agent改为纯事件驱动
- 兜底机制：24小时无事件时做一次健康检查

**计划执行：** 3/22-3/23（与按需激活同步）
