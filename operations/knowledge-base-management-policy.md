## 一、总则

### 1.1 目的

为规范梦想家乐园知识库的建设、维护和使用，确保知识资产的有效积累和传承，特制定本办法。

### 1.2 适用范围

适用于所有伙伴（阿尔法、贝塔、西塔、约塔、伽马、德尔塔、艾普西隆、艾塔、拉姆达、泽塔、卡帕）在工作过程中产生的知识性成果。

### 1.3 知识库定位

**共享知识仓库**（`/home/gang/.openclaw/shared-knowledge/`）是梦想家乐园的核心知识资产集中地，所有交付物、经验教训、决策记录、操作指南均需归档至此。

### 1.4 核心原则

- **交付即归档** — 完成任务不等于交付，文件入库才算完成
- **谁产生谁提交** — 产出者负责提交，不转嫁
- **索引即入口** — 所有知识必须可通过 index.md 找到
- **定期盘点** — 宁可多查一次，不可漏掉一个

## 二、收集机制

### 2.1 提交责任

| 知识类型 | 提交责任人 | 时限 |
|----------|-----------|------|
| 方向交付物 | 方向负责人 | 任务完成后 **24小时内** |
| 日报/简报 | 西塔（收集）/ 约塔（汇总） | 每日 **22:00前** |
| 决策记录 | 提出决策的负责人 | 决策确认后 **4小时内** |
| 故障案例 | 发现者或解决者 | 问题解决后 **24小时内** |
| 技能文档 | 伽马（技能负责人） | 技能评估/安装后 **48小时内** |
| 研究成果 | 拉姆达（研究负责人） | 研究完成后 **24小时内** |

### 2.2 提交方式

**Step 1：选择正确目录**

| 目录 | 适用内容 | 示例 |
|------|----------|------|
| `architecture/` | 系统设计、架构方案、通信协议 | 灾备方案、沟通体系设计 |
| `operations/` | 运维指南、操作流程、配置说明 | 简报流程、协作规范 |
| `cases/` | 故障排除、经验教训、最佳实践 | 心跳死锁案例 |
| `learnings/` | 学习笔记、知识总结 | SDK 补丁学习 |
| `decisions/` | 决策记录 | 知识系统设置决策 |
| `agents/` | 伙伴档案、专业领域 | 伙伴介绍 |
| `briefings/` | 日报/简报归档 | 每日简报 |

**Step 2：文件命名规范**

```
格式：{主题描述}.md
示例：
  ✅ knowledge-base-management-policy.md
  ✅ heartbeat-deadlock-lesson.md
  ✅ 2026-03-17-phase4-day1.md（简报）
  
禁止：
  ❌ doc1.md（无意义命名）
  ❌ 新建文本文档.md（默认命名）
  ❌ 文件名带空格（用连字符替代）
```

**Step 3：文件元信息**

每个文件必须包含头部元信息：

```markdown
# 文件标题

> **创建日期：** YYYY-MM-DD
> **作者：** 伙伴名 emoji
> **标签：** #标签1 #标签2
> **关联文档：** [相关文档](./path)
```

### 2.3 提交内容标准

每个交付物文件必须包含：

1. **标题** — 清晰描述内容
2. **背景/目的** — 为什么做这个
3. **具体内容** — 正文
4. **结论/行动项** — 如适用
5. **关联文档** — 相关参考

## 三、归档规范

### 3.1 目录结构

```
/home/gang/.openclaw/shared-knowledge/
├── architecture/     # 架构设计类
├── operations/       # 操作运维类
│   └── briefings/    # 简报归档
├── cases/            # 案例经验类
├── learnings/        # 学习总结类
├── decisions/        # 决策记录类
├── agents/           # 伙伴档案类
├── index.md          # 知识索引（入口）
├── CONTRIBUTING.md   # 贡献指南
├── contribution-process.md  # 贡献流程
└── README.md         # 说明文档
```

**目录新增须经行政管理部审批，个人不得擅自创建顶级目录。**

### 3.2 版本管理

- **小修订**（错字、格式）：直接修改，无需标注
- **内容更新**：修改文件头部的「最后更新」日期，保留文件名
- **重大修订**（v2.0级别）：创建新文件，文件名加版本后缀，旧文件移入 `archive/` 目录
- **废弃文档**：在文件顶部标注 `> ⚠️ 本文档已废弃，请参考 [新文档](./path)`，移入 `archive/`

### 3.3 过期/废弃文档处理

| 文档状态 | 处理方式 |
|----------|----------|
| 仍有效 | 保留在原位，定期检查更新 |
| 已过期（有新版本） | 加废弃标注，移入 `archive/` |
| 已合并到其他文档 | 加废弃标注，链接到新文档 |
| 完全无用 | 删除（需行政管理部审批） |

## 四、上传时机

### 4.1 任务完成后的归档

```
任务完成
  ↓
24小时内提交交付物到 /home/gang/.openclaw/shared-knowledge/
  ↓
更新 index.md（添加条目）
  ↓
通知行政管理部（西塔）归档完成
```

### 4.2 定期盘点

| 频率 | 责任人 | 内容 |
|------|--------|------|

| 里程碑报告 | `operations/` 相关子目录 | 方向负责人 | 里程碑达成后24h |

## 五、责任分工

### 5.1 组织架构

```
梦想家（审批）
  ↑
阿尔法（CEO，总体监督）
  ↑
西塔（行政管理部长，知识库管理总责）
  ├── 艾塔 📚（知识库日常维护、文档归档、索引更新）
  └── 泽塔 ⚙️（自动化工具开发、效率优化）
  ↑
艾塔 📚（归档监察，确保合规）
```

### 5.2 各角色职责

#### 西塔 🤝（行政管理部长）
- 知识库管理办法的制定和修订
- 协调各部门的知识归档工作
- 每日检查归档情况
- 每周组织知识库盘点
- 每月输出知识库质量报告

#### 艾塔 📚（知识库管理员）
- 日常文档归档和分类
- 维护 `index.md` 索引
- 审核提交文档的格式和分类
- 处理过期/废弃文档
- 协助伙伴查找知识

#### 泽塔 ⚙️（自动化支持）
- 开发归档自动化工具（如适用）
- 优化提交流程，降低归档门槛
- 维护归档模板

#### 各方向负责人
- 确保本方向交付物按时提交
- 审核本方向提交文档的质量
- 配合行政管理部的盘点工作

#### 艾塔 📚（归档监察）
- 每日检查各方向归档合规情况
- 发现未归档交付物，及时提醒责任人
- 连续未归档的，升级至西塔（行政管理部长）
- 参与每周盘点，独立出具检查意见

### 5.3 提交流程图

```
产出者完成交付物
  ↓
选择目录 + 命名 + 添加元信息
  ↓
提交到 /home/gang/.openclaw/shared-knowledge/ 对应目录
  ↓
更新 index.md（添加条目）
  ↓
艾塔审核（格式/分类/质量）
  ↓
通过 → 归档完成 ✅
不通过 → 反馈修改 → 重新提交
```

## 六、监督机制

### 6.1 艾塔的归档检查职责

**每日检查（自动化 + 人工）：**
- 扫描各 workspace，识别新产生的 .md 文件
- 比对 /home/gang/.openclaw/shared-knowledge/，确认是否已归档
- 未归档的，当日内提醒责任人

**检查记录格式：**

```markdown
## 知识库归档检查 — YYYY-MM-DD
检查人：艾塔 📚

| 产出者 | 交付物 | 是否已归档 | 状态 |
|--------|--------|-----------|------|
| 艾普西隆 | baseline-report.md | ✅ | 已归档 |
| 拉姆达 | xxx.md | ❌ | 未归档，已提醒 |
```

### 6.2 未按时归档的处理

| 阶段 | 时间 | 处理 |
|------|------|------|
| 第一次提醒 | 超时24h内 | 艾塔直接提醒责任人 |
| 第二次提醒 | 超时48h内 | 艾塔提醒 + 通知西塔 |
| 升级 | 超时72h | 西塔上报阿尔法，阿尔法介入 |
| 记录 | 超时72h+ | 记入月度报告，影响评估 |

### 6.3 月度审查

每月最后一个工作日，行政管理部输出《知识库月度审查报告》，内容包括：

- 本月新增文档数量和质量评估
- 归档合规率（按时归档数 / 应归档总数）
- 索引完整性检查
- 过期文档清理情况
- 下月改进计划

## 七、激励与改进

### 7.1 正向激励

- **优质贡献**：每月评选「最佳知识贡献」，在简报中表扬
- **归档模范**：连续一个月100%按时归档的，记入日志
- **知识复用**：被多次引用的文档，作者获得额外认可

### 7.2 持续改进

- 本办法每季度评估一次，根据实际运行情况修订
- 任何伙伴可提出改进建议，提交至行政管理部
- 重大修订需经梦想家审批
