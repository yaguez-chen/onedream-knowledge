> **生效：** 2026-03-17
> **维护者：** 约塔 📋（研发部长）
> **审批：** 梦想家 ✅

## 总则

本规范为梦想家乐园全体公民的工作流程标准，确保产出一致、协作高效、进度可溯。

**核心原则：**
- **文件即交付** — 所有产出必须写入文件，口头汇报不等于交付
- **进度要透明** — 每日有记录，任务有结果，阻塞有上报
- **职责要清晰** — 负责人统筹，协作人配合，跨部门协调
- **完成要归档** — 任务完成后同步到知识库，经验可复用

## 一、交付物规范

### 1.1 文件即交付原则

| 规则 | 说明 |
|------|------|
| ✅ 写入工作空间文件 | 交付物必须是工作空间中的 `.md` 文件 |
| ✅ 对话中可汇报进度 | 但最终结果必须写入文件 |
| ❌ 不接受纯口头汇报 | 仅有对话中的回复不算完成交付 |
| ❌ 不接受"我已经做了" | 必须有对应的结果文件 |

### 1.2 文件命名规范

**格式：** `{方向/类型}-{内容描述}-{日期}.md`

**示例：**
```
security-audit-baseline-2026-03-17.md     ← 安全审计基线报告
communication-norms-2026-03-17.md         ← 沟通规范
skill-inventory-2026-03-17.md             ← 技能盘点
```

**特殊文件名（固定）：**
| 文件名 | 用途 | 存放位置 |
|--------|------|----------|
| `memory/YYYY-MM-DD.md` | 每日日志 | 各自工作空间 |
| `{task-name}-result.md` | 任务完成报告 | 各自工作空间 |
| `{task-name}-blocked.md` | 阻塞报告 | 各自工作空间 |

### 1.3 文件存放位置

```
/home/gang/.openclaw/workspace-{公民名}/
├── memory/                    ← 每日日志
│   └── YYYY-MM-DD.md
├── {方向子目录}/              ← 按任务方向组织（可选）
│   └── {task-name}-result.md
├── {task-name}-result.md      ← 任务结果文件（也可放根目录）
└── ...
```

**共享知识：** 需要跨公民使用的文档 → 放入 `shared-knowledge/` 对应目录

### 1.4 交付物质量标准

- **完整性** — 包含所有要求的内容章节
- **准确性** — 信息真实，不确定的标注"待确认"
- **可读性** — 格式规范，标题清晰，列表有序
- **可追溯** — 标注日期、作者、版本

## 二、日报制度

### 2.1 每日日志要求

**每位公民每日必须产出：** `memory/YYYY-MM-DD.md`

**截止时间：** 每日 23:59（北京时间）

**模板：**

```markdown
# YYYY-MM-DD 日志

## 今日完成
- [已完成事项1]
- [已完成事项2]

## 进行中
- [进行中事项1] — 预计完成时间

## 遇到的问题
- [问题描述] — 需要谁帮助

## 明日计划
- [计划事项1]
- [计划事项2]

## 需要关注
- [需要梦想家或其他公民关注的事项]

## 状态：🟢正常 / 🟡有阻塞 / 🔴需要帮助
```

### 2.2 日报汇总流程

| 时间 | 负责人 | 动作 |
|------|--------|------|
| 每日 20:00-23:00 | 各公民 | 更新各自的 `memory/YYYY-MM-DD.md` |
| 次日 8:00-8:15 | 西塔 🤝 | 收集各公民日志 |
| 次日 8:15-8:20 | 西塔 🤝 | 对未提交公民发送提醒 |
| 次日 8:20-8:45 | 西塔 🤝 | 整理分析，生成每日简报 |
| 次日 8:45-9:00 | 西塔 🤝 | 存储简报，通知梦想家 |

**简报存储：** `shared-knowledge/operations/briefings/YYYY-MM-DD.md`

### 2.3 未提交处理

| 情况 | 处理 |
|------|------|
| 8:15 未提交 | 西塔发送提醒 |
| 8:45 仍未提交 | 简报中标记"未提交" |
| 连续2天未提交 | 贝塔介入检查原因 |
| 连续3天未提交 | 升级到阿尔法处理 |

## 三、任务确认制度

### 3.1 任务完成标准

**任务完成必须满足：**
1. ✅ 产出对应的结果文件（`{task-name}-result.md`）
2. ✅ 结果文件包含：完成内容、交付物列表、备注说明
3. ✅ 负责人确认（方向负责人审核通过）
4. ✅ 文件存放在正确位置

**结果文件模板：**

```markdown
# 任务完成报告：{任务名称}

**任务编号：** [如有]
**负责人：** [姓名 emoji]
**协作人：** [姓名 emoji，如有]
**开始时间：** YYYY-MM-DD
**完成时间：** YYYY-MM-DD

## 完成内容
- [完成项1]
- [完成项2]

## 交付物
| 交付物 | 路径 | 说明 |
|--------|------|------|
| [文件名] | [完整路径] | [简述] |

## 备注
- [如有特殊情况说明]

## 后续建议
- [如有改进建议或后续工作]
```

### 3.2 贝塔每日检查机制

**检查时间：** 每日 9:00（北京时间）

**检查内容：**
- [ ] 各公民工作空间是否有新的产出文件
- [ ] 各公民前一日的日志是否已提交
- [ ] 各任务是否有对应的完成报告

**检查结果：** 更新到贝塔的 `memory/YYYY-MM-DD.md` 中

**异常处理：**
| 情况 | 处理 |
|------|------|
| 当日无产出 | 记录并提醒该公民 |
| 连续2天无产出 | 升级到阿尔法处理 |
| 任务延期超过2天 | 要求提交阻塞报告 |
| 发现安全问题 | 立即上报梦想家 |

### 3.3 阻塞报告

**当任务无法按时完成时，必须提交：** `{task-name}-blocked.md`

```markdown
# 阻塞报告：{任务名称}

**阻塞时间：** YYYY-MM-DD HH:MM
**预计影响：** 延期 X 天

## 阻塞原因
- [具体原因]

## 已尝试的解决方法
- [尝试1] — 结果：[失败/部分成功]
- [尝试2] — 结果：[失败/部分成功]

## 需要的帮助
- [需要谁/什么资源来解除阻塞]

## 建议方案
- [如果无法解除，建议的替代方案]
```

## 四、协作规范

### 4.1 汇报关系

根据[组织架构 v1.1](../architecture/organizational-structure.md)：

```
梦想家（董事长）
  ↑ 双线汇报
阿尔法（CEO）← 贝塔/西塔/约塔（部长级，双线汇报）
  ├── 贝塔（监察安全部）→ 贝塔 + 艾普西隆
  ├── 西塔（行政管理部）→ 西塔 + 泽塔 + 艾塔
  └── 约塔（研发部）→ 约塔 + 伽马 + 德尔塔 + 拉姆达 + 卡帕
```

### 4.2 任务分配与执行

**负责人 = 最终交付责任人**

| 角色 | 职责 |
|------|------|
| **方向负责人** | 统筹规划、任务分配、质量把控、最终确认交付 |
| **协作人** | 配合执行、完成分配的任务、及时汇报进展 |
| **约塔（跨部门协调）** | 跨方向任务协调、资源调配、进度仲裁 |

**关键规则：**
- 交付物由**负责人确认后**才算完成
- 协作人发现无法完成时，**2小时内上报**
- 负责人发现延期风险时，**立即通知阿尔法**

### 4.3 跨部门协作流程

**启动条件：** 任务涉及多个部门或方向

**流程：**
```
1. 提出方联系约塔说明需求
   ↓
2. 约塔评估任务范围和资源需求
   ↓
3. 约塔与相关部门负责人协调
   ↓
4. 约塔创建任务并分配
   ↓
5. 各协作方执行，约塔跟踪进度
   ↓
6. 约塔汇总交付物，通知提出方
```

**协作请求格式（通过 sessions_send 或文件传递）：**

```markdown
🤝 协作请求

来自：[姓名 emoji]
主题：[简短描述]
背景：[为什么需要协作]
需求：[具体需要什么帮助]
时间线：[截止时间]
优先级：🔴高 / 🟡中 / 🟢低
相关文档：[路径/链接]
```

### 4.4 协作响应时效

| 收到请求后 | 行动 |
|-----------|------|
| 15分钟内 | 确认收到（哪怕只是"收到，稍后看"） |
| 2小时内 | 给出初步回应（能做/不能做/需要更多信息） |
| 按截止时间 | 完成或更新进度 |

### 4.5 与梦想家沟通

| 情况 | 方式 |
|------|------|
| 紧急安全事件 | 直接联系梦想家 + 通知西塔备案 |
| 日常进度报告 | 通过西塔汇总（每日简报） |
| 需要梦想家决策 | 通过西塔整理选项后提交 |
| 梦想家询问你 | 直接回复 |
| 非紧急问题/建议 | 通过西塔汇总 |

### 4.6 冲突解决

| 冲突类型 | 处理方式 |
|----------|----------|
| 资源冲突 | 约塔仲裁 |
| 技术分歧 | 列出方案优缺点，约塔或梦想家裁决 |
| 流程分歧 | 西塔协调，必要时梦想家决定 |
| 优先级分歧 | 约塔根据整体情况判断，必要时阿尔法裁定 |

**升级路径：**
```
公民间无法解决 → 约塔/西塔协调
仍无法解决 → 阿尔法裁定
涉及重大决策 → 提报梦想家
```

## 五、归档流程

### 5.1 归档原则

- 所有已完成的任务交付物应归档到 `shared-knowledge/`
- 归档后的文档供所有公民查阅和复用
- 经验教训应记录到 `shared-knowledge/learnings/`

### 5.2 归档时机

| 触发条件 | 归档动作 |
|----------|----------|
| 任务完成 | 交付物归档到 `shared-knowledge/` 对应目录 |
| 项目结束 | 项目报告 + 经验教训归档 |
| 发现重要教训 | 立即记录到 `learnings/` |
| 有通用价值的文档 | 归档到 `shared-knowledge/` 对应分类 |

### 5.3 归档路径

```
shared-knowledge/
├── architecture/      ← 系统设计、架构方案
├── operations/        ← 流程规范、操作指南
├── cases/             ← 问题解决、最佳实践
├── agents/            ← 伙伴档案
├── decisions/         ← 决策记录
├── learnings/         ← 经验教训
├── templates/         ← 文档模板
└── index.md           ← 知识索引（每次归档后更新）
```

### 5.4 归档流程

```
1. 完成任务 → 产出结果文件
   ↓
2. 负责人确认质量
   ↓
3. 复制/移动到 shared-knowledge/ 对应目录
   ↓
4. 更新 shared-knowledge/index.md（添加条目）
   ↓
5. 通知相关公民（如有必要）
```

### 5.5 索引更新要求

每次归档文档后，必须更新 `shared-knowledge/index.md`：

```markdown
| 文件 | 描述 | 标签 | 最后更新 |
|------|------|------|----------|
| `新文件名.md` | 简短描述 | #标签1 #标签2 | YYYY-MM-DD |
```

**同时更新统计数字：**
```markdown
- **总文档数：** [新总数]
```

## 六、制度执行保障

### 6.1 监督机制

| 监督者 | 范围 | 频率 |
|--------|------|------|
| 贝塔 🔵 | 各公民产出检查 | 每日 |
| 约塔 📋 | 任务进度跟踪 | 每日 |
| 西塔 🤝 | 日报收集与汇总 | 每日 |
| 阿尔法 🦐 | 整体执行情况 | 每周 |

### 6.2 违规处理

| 情况 | 处理 |
|------|------|
| 未按时提交日报 | 西塔提醒 → 贝塔检查 → 升级阿尔法 |
| 任务未写结果文件 | 贝塔要求补交 |
| 连续2天无产出 | 贝塔上报阿尔法 |
| 连续3天无产出 | 阿尔法直接介入 |

### 6.3 制度修订

- 本规范由约塔维护
- 任何公民可提出修订建议
- 重大修订需阿尔法审核、梦想家批准
- 修订记录记录在本文档底部
